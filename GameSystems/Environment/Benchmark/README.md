# Snow의 mesh vs RT 성능 비교

적설 방식을 mesh에서 RT로 바꾼 핵심 이유는 **CPU 부하**였습니다. 그 주장을 숫자로 검증하기 위해, 동일 조건에서 두 방식의 성능을 직접 측정했습니다.

이 문서는 **무엇을 · 왜 · 어떻게 측정했는지**를 정리합니다. 측정 코드는 [`PerformanceProbe.cs`](./PerformanceProbe.cs) 입니다.

---

## 무엇을 측정했나

핵심 주장이 "mesh는 **CPU**(특히 메인 스레드)가 병목"이므로, fps만으로는 부족합니다. fps는 해상도·GPU 상황에 휘둘려 CPU 문제를 가릴 수 있기 때문입니다. 그래서 **메인 스레드 프레임 시간**을 1순위 지표로 삼았습니다.

| 지표 | 의미 | 왜 보는가 |
|---|---|---|
| **Main thread time** (ms) | 한 프레임의 메인 스레드 점유 시간 | CPU 병목 여부를 직접 보여줌 (핵심) |
| **Frame time** (ms) | 전체 프레임 시간 | 체감 성능 |
| **Avg FPS** | 평균 프레임률 | 참고용 |

각 지표는 **평균 / 중앙값 / p95 / 최대**로 요약합니다. 평균만 보면 순간 스파이크가 묻히므로, **p95(상위 5% 최악 프레임)** 를 함께 봐서 스터터링을 잡아냅니다.

---

## 어떻게 측정했나

Unity의 **`ProfilerRecorder`** 로 메인 스레드 시간을 샘플링하고, 프레임 시간은 `Time.unscaledDeltaTime`으로 수집합니다.

```csharp
// "Main Thread" 카운터는 Internal 카테고리에 ns 단위로 노출됨 → CPU 병목 검증의 핵심
mainThreadRecorder = ProfilerRecorder.StartNew(ProfilerCategory.Internal, "Main Thread");
...
frameMs.Add(Time.unscaledDeltaTime * 1000f);
if (mainThreadRecorder.Valid)
    mainThreadMs.Add(mainThreadRecorder.LastValue * 1e-6f); // ns → ms
```

측정이 끝나면 평균/중앙값/p95/최대를 콘솔에 출력하고 **CSV로 저장**합니다. 같은 CSV에 append하므로 mesh·RT 두 번의 실행 결과가 한 파일에 쌓여 바로 비교됩니다.

---

## 공정한 비교를 위한 조건

숫자가 신뢰받으려면 **mesh/RT 외의 변인을 통제**해야 합니다.

- **측정 환경 고정** — 절대 수치는 빌드가 이상적이지만(에디터는 오버헤드가 큼), 이번 비교는 mesh/RT를 **동일하게 에디터에서** 측정해 상대 차이를 봅니다.
- **워밍업 제외** — 초기 셰이더 컴파일·풀 생성 스파이크를 빼기 위해 첫 `warmupSeconds`(기본 3초)는 수집하지 않습니다.
- **거대 프레임 제외** — 에디터 멈춤·컴파일·포커스 아웃은 한 프레임이 수십 초로 잡힙니다. `maxValidFrameSeconds`(기본 0.5초)를 넘는 프레임은 측정 시간 누적·샘플에서 모두 제외해, 측정이 조기 종료되거나 max가 오염되지 않게 합니다.
- **동일 측정 구간** — 두 방식 모두 `measureSeconds`(기본 30초) 동안 수집합니다.
- **동일 씬·카메라·적설 범위** — mesh/RT만 다르고 나머지는 같게 둡니다.
- **방식 전환** — 두 방식은 재실행으로 전환합니다(종료 → 모드 전환 → 재실행). 프로브는 현재 실행 중인 방식이 무엇이든 그 성능을 측정하므로, 측정 절차 자체는 동일합니다.

## 측정 결과

> **에디터 측정** 기준입니다. 에디터는 오버헤드가 있어 절대 수치는 빌드와 다를 수 있으나, mesh ↔ RT **상대 비교의 경향성**은 유효합니다.

| 지표 | mesh | RT | 개선 |
|---|---|---|---|
| **Main thread avg** (ms) | 25.67 | 19.45 | **−24%** |
| Main thread median (ms) | 21.10 | 18.56 | −12% |
| Main thread p95 (ms) | 35.10 | 25.86 | −26% |
| **Main thread max** (ms) | 456.34 | 118.97 | **−74%** |
| Frame avg (ms) | 25.68 | 19.47 | −24% |
| **Avg FPS** | 38.9 | 51.4 | **+32%** |

측정 프레임 수: mesh 1170 / RT 1542 (각 30초, 워밍업·거대 프레임 제외 후).

**해석**

- 두 방식 모두 **Main thread time ≈ Frame time** 입니다(mesh 25.67≈25.68, RT 19.45≈19.47). 프레임이 GPU가 아니라 **메인 스레드(CPU)를 기다리고 있다**는 뜻으로, "CPU가 병목"이라는 전제가 측정으로 확인됩니다.
- mesh → RT 전환으로 **메인 스레드 부하를 약 6.2ms(−24%) 덜어냈고**, 이것이 그대로 **FPS +32%** 로 이어졌습니다. 눈 처리를 GPU로 옮긴 효과가 메인 스레드에서 직접 빠진 것입니다.
- **최악 프레임(max) 456 → 119ms (−74%)** — mesh의 큰 스터터(메시 재생성 스파이크로 추정)가 RT에서 크게 줄어, 평균보다 체감 차이를 더 잘 보여줍니다.
- 다만 RT도 여전히 메인 스레드 ≈ 프레임이므로, RT가 CPU 병목을 *완전히* 없앤 것은 아닙니다. 눈 부분의 CPU 비용을 덜어 **병목을 완화**한 것이며, 전체적으로는 아직 메인 스레드 바운드입니다.

> 측정 환경: AMD Ryzen 9 3900X 12-Core · NVIDIA GeForce RTX 2080 Ti · 1209×606 (에디터 Game 뷰) · Windows 10 · Unity 6000.0.27f1 · 측정 방식: 에디터

## 한계 / 메모

- `ProfilerRecorder`의 "Main Thread" 카운터는 Unity 버전에 따라 노출명이 다를 수 있습니다. 값이 0으로 나오면 카운터명을 확인하세요.
- 더 정밀하게 **눈 갱신 구간만** 떼어 보려면, 해당 메서드를 `ProfilerMarker`로 감싸 별도 Recorder로 측정할 수 있습니다 (이 프로브는 메인 스레드 전체를 봅니다).
